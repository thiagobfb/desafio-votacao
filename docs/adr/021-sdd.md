# ADR-021: Adoção de Spec Driven Development (SDD)

- **Status:** Aceito
- **Data:** 2026-04-30
- **Contexto:** Decisão metodológica do projeto

## Contexto

O avaliador precisa entender **por quê** das escolhas, não só **o quê** foi feito. Sem rastreabilidade explícita, decisões viram folclore — quem chega depois "descobre" sozinho.

## Decisão

Estrutura SDD inspirada em **Spec Kit** (GitHub):

```
specs/
├── constitution.md                # 8 princípios invariantes
├── README.md                      # como SDD funciona aqui
└── 001-sistema-votacao/
    ├── spec.md                    # O QUÊ — RFs, RNs, fluxos, escopo
    ├── plan.md                    # COMO — stack, arquitetura, contratos
    └── tasks.md                   # passo-a-passo executável
```

Cada feature ganha pasta numerada com o trinômio **spec → plan → tasks**. Decisões técnicas que cabem em uma seção do plan ficam lá; decisões transversais ou que afetam múltiplas features viram **ADR** em `docs/adr/`.

## Consequências

**Prós:**
- Rastreabilidade: cada commit pode referenciar `T-NNN` da `tasks.md`.
- Onboarding leve — `specs/` conta a história.
- Evita "design por commit message" (decisões enterradas em git log).

**Trade-offs:**
- Overhead se virar burocracia. Mitigamos mantendo specs curtas e específicas; este projeto inteiro é uma única spec (001).

## Alternativas consideradas

- **Apenas issues/PRs no GitHub:** mais leve, perde contexto offline.
- **Wiki externa:** drift do código (atualiza-se um, esquece-se o outro).
- **ADRs sem spec/plan:** captura decisões mas não captura **requisitos** (RFs/RNs).
